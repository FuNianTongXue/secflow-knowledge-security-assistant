# Native Java Finance CFG/DFG Engine

## Scope

The engine is an independent Tree-sitter-based implementation. Its architecture follows public concepts visible in the CodeQL Java libraries, including basic blocks, node-order dominance, guards, def-use flow, callable summaries, and type-aware dispatch. No CodeQL implementation code is copied or embedded.

The native finance query is intentionally separate from the generic taint queries and from Semgrep. Semgrep remains a fast discovery/fallback layer; it is not used as ground truth for native-engine metrics.

## Pipeline

1. Parse Java classes, methods, annotations, parameters, fields, statements, and calls with Tree-sitter.
2. Build a statement-level CFG for each method, including conditions, loops, switch nodes, try/catch/finally paths, returns, and throws.
3. Compute a dominator set for every CFG node. Calls in one block are additionally ordered by source byte offset.
4. Build idempotency-key def-use edges through parameters, request headers, DTO getters, assignments, constructor arguments, and call arguments.
5. Extract typed finance effects:
   - durable atomic claim;
   - lookup-only/check-then-act guard;
   - lock-only non-durable guard;
   - result persistence/replay;
   - database money mutation;
   - external payment-gateway effect;
   - state read and compare-and-set transition;
   - transaction boundary and storage domain.
6. Resolve same-class, concrete receiver, explicit-constructor, and unique interface-to-implementation calls.
7. Propagate effect summaries to callers by fixed point while remapping callee parameter origins to caller argument origins.
8. Query unsafe HTTP/message entrypoints and require a request-derived atomic durable claim to dominate every money effect.

## Native Queries

| Scenario | Required proof |
|---|---|
| Missing idempotency | A replayable entry reaches a money effect with no request-derived atomic durable claim dominating it. |
| Check then act | A lookup dominates the money effect, but no atomic claim does. |
| Guard after effect | The money effect dominates/precedes the valid claim. |
| Non-durable guard | A process/distributed lock is present without durable replay state. |
| Missing transaction | A database/memory money mutation is not enclosed by a propagated transaction boundary. |
| Cross-store consistency | The claim and money write use different stores without a persisted completion/result path. |
| State transition race | A status read dominates a money mutation without a compare-and-set/state-transition effect. |

## Evidence Contract

Every native finding contains:

- `engine = java-native-finance`;
- a source, sink, and interprocedural path;
- `cfg_graph` node/edge counts and a dominance result;
- `dfg_edges` for the idempotency key;
- `semantic_proof` containing risk kind, transaction state, store, and interprocedural depth.

## Evaluation

Run the labeled regression:

```bash
.venv/bin/python scripts/evaluate_java_finance_native.py
```

The current evaluation has 33 independently labeled semantic templates and 64 deterministic symbol/format variants per template. The 2,112 variants test parser and flow stability but remain correlated; only the 33 template labels are independent units.

Run the real-project upload test:

```bash
.venv/bin/python scripts/evaluate_upload_idempotency.py \
  --server-url http://127.0.0.1:18782 \
  --output docs/github-finance-trading-java-200-native-results.json \
  --markdown docs/github-finance-trading-java-200-native-2026-07-19.md
```

The GitHub set validates cloning, upload limits, runtime stability, and report generation. It has no maintainer-provided line-level ground truth, so it must not be used to claim Precision, Recall, FPR, or FNR.

## Current Limits

- The DFG tracks value origins and def-use edges but does not yet construct full SSA phi nodes.
- Exception edges are conservative; checked/unchecked exception type semantics are not modeled separately.
- Virtual dispatch supports unique concrete and common Spring `*Impl` mappings, not arbitrary points-to analysis.
- Reflection, runtime proxies, generated bytecode, native methods, and cross-service network calls require external models.
- Redis/database recovery is accepted only when a durable completion/result path is visible; full distributed protocol verification requires deployment topology and storage configuration.
- Accuracy claims beyond the labeled corpus require independent manual labels on real projects and a held-out project-level split.
